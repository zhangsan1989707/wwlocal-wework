## 第 3 页 | 系统概述与目标

**解决什么问题？**

- 政务微信开放接口返回的业务数据经过 RSA+AES 双层加密
- 需要一个平台统一解密、存储、查询和审计这些数据

**核心能力**

-  自动从政务微信 API 拉取加密日志
-  RSA+AES 双层解密后存储到本地 MySQL
-  多维度查询、筛选、导出
-  组织架构同步与人员关联
-  操作审计日志自动记录

## 第 10 页 | 数据同步 - 定期同步

**左侧菜单 → 同步任务**

**定期同步（自动调度）**：

- 点击"启动"按钮开启定时自动增量同步
- 同步间隔可配置：10 分钟 / 30 分钟 / 1 小时
- 点击"停止"暂停调度
- 状态面板显示：运行状态、间隔、上次/下次执行时间

**增量同步原理**：

- 系统记录每次同步的断点（`last_log_time`）
- 下次同步从断点 +1 开始，只拉取新数据
- 即使中间有数据未同步完成，下次仍会从断点继续，不丢失数据

**操作**：现场演示启动/停止调度

## 第 14 页 | 密钥管理

**左侧菜单 → 密钥管理**

**密钥列表**：展示所有 RSA 密钥版本及状态（活跃/备用）

**核心操作**：

| 操作 | 说明 |
|------|------|
| 添加密钥 | 输入版本号 + PEM 内容，上传新的 RSA 私钥 |
| 切换活跃 | 将备用密钥设为活跃，后续解密优先使用 |
| 测试密钥 | 验证密钥格式是否正确（位数、类型） |

**安全机制**：

- PEM 文件存储在 `keys/{version}/rsa_private_key.pem`
- 系统启动自动检查权限，宽松权限自动修正为 600
- 解密失败时自动回退到历史版本密钥

**操作**：现场演示添加和切换密钥

## 第 20 页 | 常见操作流程 - 故障排查

| 现象 | 排查步骤 |
|------|---------|
| 同步卡死 | 同步任务 → 取消 → 重新启动；检查后端日志 `docker-compose logs backend` |
| 解密失败 | 密钥管理 → 测试密钥 → 确认密钥版本正确且已激活 |
| 查询无结果 | 确认时间范围正确；确认数据类型已启用且已同步 |
| 登录被锁定 | 等待 1 分钟自动解锁；检查是否有人在尝试暴力破解 |
| 前端无法访问 | 检查 `docker-compose ps`；检查端口 18073 是否被占用 |
| 政务微信连接失败 | 检查 `WEWORK_BASE_URL`；自签证书需设 `INSECURE_SKIP_VERIFY=true` |

## 第 21 页 | 关键概念速查

| 概念 | 说明 |
|------|------|
| Feature ID | 政务微信定义的数据类型编号，每个编号对应一种业务日志 |
| 增量同步 | 从上次断点继续拉取新数据，不重复拉取历史数据 |
| 全量同步 | 按指定时间范围重新拉取数据，适合数据修复 |
| 密钥版本 | RSA 密钥的版本标识，支持多版本热切换 |
| openid | 手机号的加密标识，在本系统中即手机号 |
| enc_data_hash | 加密数据的 MD5 哈希，用于去重（INSERT IGNORE） |

## 第 22 页 | Q&A

**常见问题**：

**Q：同步的数据周期是多久？**
A：定时调度默认 1 小时执行一次增量同步，每次拉取从上次断点到当前时间的新数据。

**Q：密钥更换会影响历史数据吗？**
A：不会。密钥切换只影响后续新数据的解密，已解密的数据不受影响。解密失败时系统会自动尝试历史密钥。

**Q：如何知道哪些人没有使用政务微信？**
A：总览看板 → 未使用人员分析，设置天数阈值即可筛选。

**Q：数据会重复存储吗？**
A：不会。系统使用 `enc_data_hash`（MD5）做去重，即使重复同步也不会产生重复记录。
