# 运维手册

## 1. 服务管理

### 启动服务

```bash
docker-compose up -d --build
```

### 停止服务

```bash
docker-compose down
```

### 重启服务

```bash
docker-compose restart
```

### 查看服务状态

```bash
docker-compose ps
```

正常运行时，三个容器（mysql、backend、frontend）状态应为 `Up`，且 mysql 的 healthcheck 显示 `healthy`。

### 查看日志

```bash
# 查看后端日志（实时跟踪）
docker-compose logs -f backend

# 查看前端（nginx）日志
docker-compose logs -f frontend

# 查看 MySQL 日志
docker-compose logs -f mysql

# 查看最近 100 行后端日志
docker-compose logs --tail=100 backend
```

### 重启单个服务

```bash
docker-compose restart backend
docker-compose restart frontend
docker-compose restart mysql
```

### 端口映射

| 服务 | 容器名 | 宿主机端口 | 容器端口 |
|------|--------|-----------|---------|
| MySQL | wework-mysql | 3307 | 3306 |
| 后端 | wework-backend | 3010 | 8080 |
| 前端 | wework-frontend | 5173 | 80 |

## 2. 健康检查

### 检查命令

```bash
curl http://localhost:3010/health
```

### 正常响应（HTTP 200）

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "status": "ok",
    "checks": {
      "db": "ok",
      "keys": "ok"
    }
  }
}
```

### 异常响应（HTTP 503）

```json
{
  "status": "unhealthy",
  "checks": {
    "db": "error",
    "keys": "ok"
  }
}
```

`checks.db` 或 `checks.keys` 任一为非 `ok` 状态时，返回 HTTP 503。

### 检查项说明

| 检查项 | 含义 | 异常时排查方向 |
|--------|------|---------------|
| db | MySQL 连接和 Ping | 检查 MySQL 容器状态、网络、.env 中 DB_PASSWORD |
| keys | 密钥目录可读性 | 检查 keys/ 目录是否存在、容器挂载是否正确 |

## 3. 数据备份

### MySQL 备份

```bash
docker exec wework-mysql mysqldump -u root -p wework_logs > backup_$(date +%Y%m%d).sql
```

提示输入密码时，填写 `.env` 文件中 `DB_PASSWORD` 的值。

### MySQL 恢复

```bash
docker exec -i wework-mysql mysql -u root -p wework_logs < backup_20260514.sql
```

恢复前建议先停止后端服务，避免写入冲突：

```bash
docker-compose stop backend
# 执行恢复
docker-compose start backend
```

### 密钥文件备份

```bash
tar czf keys_backup_$(date +%Y%m%d).tar.gz keys/
```

`keys/` 目录包含所有 PEM 私钥文件，丢失后无法解密历史数据，务必妥善保管。

### 备份建议

- 每日定时备份 MySQL 数据和 keys 目录
- 备份文件应存放在与服务器不同的物理位置
- 定期验证备份文件可正常恢复
- 建议保留至少 30 天的备份历史

## 4. 密钥文件管理

### 文件位置

```
keys/{version}/rsa_private_key.pem
```

例如：`keys/v1/rsa_private_key.pem`、`keys/v2/rsa_private_key.pem`

### 权限要求

PEM 文件权限应为 `0600`（仅所有者可读写）。系统启动时会自动检测并修正过宽的权限设置。

### 新增密钥

1. 将 PEM 文件放入 `keys/{version}/` 目录：

```bash
mkdir -p keys/v2
cp /path/to/new_rsa_private_key.pem keys/v2/rsa_private_key.pem
chmod 600 keys/v2/rsa_private_key.pem
```

2. 通过前端"密钥管理"页面添加密钥记录，填写版本号和 PEM 内容。

3. 添加后密钥默认为非激活状态，需要手动激活才会用于后续解密。

### 切换密钥

在前端"密钥管理"页面激活目标版本的密钥。

注意事项：
- 密钥切换仅影响后续解密操作，不会修改已解密的历史数据
- 解密失败时，系统会自动遍历所有历史密钥版本尝试回退解密
- 建议在切换前使用"测试密钥"功能验证新密钥可用

### 密钥轮换建议

- 密钥使用超过 90 天会在看板页面显示告警
- 建议定期轮换密钥以提高安全性
- 轮换流程：生成新密钥 → 放入 keys 目录 → 前端添加 → 测试通过 → 激活

## 5. 扩容建议

### MySQL 调优

`innodb_buffer_pool_size` 建议设为可用内存的 70-80%。修改 `mysql/my.cnf` 中的配置后重启 MySQL 容器。

### 大数据量场景

Feature `90000066`（API 调用日志）每天可能产生 30 万条以上记录，针对此类高频数据类型：
- 可考虑在前端禁用该 feature 的同步，仅在需要时手动触发
- 关注 `log_*` 表大小，必要时归档历史数据

### 磁盘空间估算

```
所需磁盘 ≈ 每天数据量（MB）× 保留天数 + MySQL 索引开销（约 30%）+ 50MB（系统表）
```

可通过 `GET /api/v1/system/status` 接口查看当前各表的实际大小。

### 数据保留策略

建议根据业务需求定期清理过期的月表：

```sql
-- 示例：删除 6 个月前的月表
DROP TABLE IF EXISTS log_90000031_202511;
```

清理前确保已备份。

## 6. 故障排查

### 同步卡死

**现象：** 点击同步后长时间无响应，状态一直显示"同步中"。

**排查步骤：**

1. 查看后端日志是否有错误：
```bash
docker-compose logs backend | grep -i error
```

2. 如果没有明显错误，尝试重启后端服务：
```bash
docker-compose restart backend
```

3. 重启后同步状态会重置，可重新发起同步。

### 解密失败

**现象：** 同步完成但部分或全部记录解密失败。

**排查步骤：**

1. 检查当前激活的密钥版本是否与政务微信后台配置一致
2. 在前端"密钥管理"页面使用"测试密钥"功能验证密钥
3. 检查 PEM 文件内容是否完整（应包含 `-----BEGIN RSA PRIVATE KEY-----` 和 `-----END RSA PRIVATE KEY-----`）
4. 如果密钥已更换，将新旧版本都添加到系统中，解密时会自动回退尝试

### 数据库连接失败

**现象：** `/health` 返回 `"db": "error"`。

**排查步骤：**

1. 检查 MySQL 容器状态：
```bash
docker-compose ps mysql
```

2. 检查 `.env` 文件中 `DB_PASSWORD` 是否正确

3. 查看 MySQL 日志：
```bash
docker-compose logs mysql
```

4. 如果 MySQL 容器反复重启，检查磁盘空间和内存是否充足

### 政务微信 API 连接失败

**现象：** 同步任务报错，日志中出现网络相关错误。

**排查步骤：**

1. 检查 `.env` 文件中的配置：
   - `WEWORK_BASE_URL`：API 地址是否正确
   - `WEWORK_CORPID`：企业 ID 是否正确
   - `WEWORK_SECRET`：应用密钥是否正确

2. 测试网络连通性：
```bash
docker exec wework-backend curl -v ${WEWORK_BASE_URL}
```

3. 如果使用自签名证书，设置 `WEWORK_INSECURE_SKIP_VERIFY=true`

### 前端无法访问

**现象：** 浏览器无法打开 `http://<host>:5173`。

**排查步骤：**

1. 检查前端容器状态：
```bash
docker-compose ps frontend
```

2. 检查宿主机防火墙是否放行了 5173 端口

3. 检查前端容器日志：
```bash
docker-compose logs frontend
```

4. 确认后端服务正常（前端依赖后端 API）

## 7. 监控告警

### 建议监控项

| 监控项 | 方法 | 频率 | 告警条件 |
|--------|------|------|---------|
| 服务存活 | `GET /health` | 每分钟 | HTTP 非 200 或返回 `status != "ok"` |
| 数据新鲜度 | `GET /api/v1/system/status` | 每小时 | `data_age_hours > 48` |
| 同步成功率 | `GET /api/v1/sync-history` | 每小时 | `failed > 0` |
| 密钥状态 | `GET /api/v1/dashboard/overview` | 每天 | `active_key_days > 90` |
| 通讯录同步 | `GET /api/v1/system/status` | 每天 | `contacts.sync_age_hours > 168`（7 天） |
| 磁盘空间 | 系统监控 | 每天 | 剩余空间低于 20% |

### 简易监控脚本示例

```bash
#!/bin/bash
# health_check.sh - 建议配合 crontab 每分钟执行

RESPONSE=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:3010/health)
if [ "$RESPONSE" != "200" ]; then
  echo "[ALERT] 健康检查失败，HTTP 状态码: $RESPONSE" | mail -s "服务告警" admin@example.com
fi
```

### 看板问题提醒

`GET /api/v1/dashboard/overview` 返回的 `problems` 数组包含自动检测的问题：

| 问题 | 级别 | 触发条件 |
|------|------|---------|
| 最近一次同步有失败记录 | error | 最新同步任务 `failed > 0` |
| 当前密钥版本已使用超过 90 天 | warning | 激活密钥 `activated_at` 距今超过 90 天 |
| 通讯录超过 7 天未同步 | warning | 最近一次通讯录同步距今超过 7 天 |
| 最近 24 小时无新日志入库 | warning | 所有 feature 的 `last_log_time` 均在 24 小时之前 |
