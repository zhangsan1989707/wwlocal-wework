# 政务微信数据审计平台 - 系统架构设计

## 1. 系统概述

政务微信数据审计平台是一个本地化部署的数据采集与审计系统。核心功能是通过政务微信开放接口获取加密的业务日志，经过 RSA + AES 双层解密后存储到本地 MySQL 数据库，前端提供日志查询、数据同步、通讯录管理、密钥管理等功能。

**设计原则**：
- 数据不出内网：所有组件部署在本地服务器，不依赖外部服务
- 最小权限：密钥文件自动检查并修正权限（600）
- 单账号体系：目前仅支持单管理员账号，无 RBAC

## 2. 整体架构

系统由三个 Docker 容器组成：

```
                    用户浏览器
                        │
                        ▼
              ┌──────────────────┐
              │   Nginx (:80)    │
              │  前端静态资源      │
              │  反向代理 /api/   │
              └────────┬─────────┘
                       │ /api/* → proxy_pass
                       ▼
              ┌──────────────────┐
              │  Go Backend      │
              │  Echo (:8080)    │
              └────────┬─────────┘
                       │ GORM
                       ▼
              ┌──────────────────┐
              │   MySQL 8.0      │
              │  (:3306 内部)     │
              └──────────────────┘
```

**端口映射（宿主机）**：

| 服务 | 容器内端口 | 宿主机端口 |
|------|-----------|-----------|
| MySQL | 3306 | 13307 |
| 后端 Go | 8080 | 19010 |
| 前端 Nginx | 80 | 18073 |

**Nginx 反向代理配置**：
- `/api/` 路径代理到 `http://backend:8080`
- 其他路径回退到 `index.html`（支持 Vue SPA 路由）
- `proxy_read_timeout` 和 `proxy_send_timeout` 设为 300s，适应长时间同步请求

**依赖关系**：后端 `depends_on` MySQL（`condition: service_healthy`），前端 `depends_on` 后端。

## 4. 依赖注入

`cmd/server/main.go` 是唯一的组合根，采用手动构造注入，不使用任何 DI 框架。

**注入链路**：

```
Config (config.yaml + env vars)
  │
  ▼
GORM DB (MySQL 连接池: maxOpen=50, maxIdle=25, maxLifetime=5min)
  │
  ├── KeyRepository (db, keysDir, encryptKey)
  ├── LogRepository (db)
  ├── SyncStateRepository (db)
  ├── ContactRepository (db)
  ├── SyncFeatureRepository (db)
  ├── SyncHistoryRepository (db)
  └── OperationLogRepository (db)
  │
  ▼
Services
  ├── WeWorkService (config.WeWork)
  ├── DecryptService (KeyRepository)
  ├── SyncService (WeWorkService, DecryptService, LogRepository, KeyRepository,
  │                SyncStateRepository, SyncHistoryRepository, SyncFeatureRepository, Config)
  ├── QueryService (LogRepository, ContactRepository, WeWorkService, DecryptService,
  │                SyncFeatureRepository, Config)
  ├── ContactService (config.WeWork)
  ├── ContactSyncService (ContactService, ContactRepository, SyncHistoryRepository)
  ├── SchedulerService (SyncService, interval)
  ├── KeyService (KeyRepository)
  └── OperationLogService (OperationLogRepository)
  │
  ▼
Handlers（每个 handler 接收对应的 service）
  │
  ▼
Router (所有 handler + OperationLogService + jwtSecret)
  │
  ▼
Echo server
```

**启动时初始化**：
1. 加载配置（yaml + 环境变量覆盖）
2. 建立数据库连接池
3. 各 Repository 执行 AutoMigrate
4. 从 config.yaml 初始化 sync_features 表
5. 校验 sync_state 与实际数据的一致性
6. 检查密钥文件权限（递归，修正过于宽松的权限为 600）
7. 若配置 `scheduler.enabled: true`，自动启动定时调度

**优雅关闭顺序**：
1. 停止定时调度
2. 取消正在进行的同步任务
3. 优雅关闭 HTTP（最多等待 30 秒）
4. 关闭数据库连接

## 5. 核心流程

### 5.1 数据同步

```
SyncService.SyncFeature(featureID, startTime, endTime)
  │
  ├── 按天拆分时间范围（政务微信 API 要求 start_time 和 end_time 在同一天）
  │
  └── 对每一天:
        │
        ├── WeWorkService.GetLogList(featureID, dayStart, dayEnd, limit)
        │     │
        │     ├── 获取 access_token（先查缓存，过期则重新请求）
        │     └── 调用政务微信 API，分页拉取（每页最多 1000 条）
        │
        ├── 对每条记录:
        │     │
        │     └── DecryptService.Decrypt(item)
        │           │
        │           ├── RSA PKCS1v15 解密 encrypt_key (base64) → 16 字节 AES 密钥
        │           ├── AES-128-CBC 解密 encrypt_data (base64)，IV = 密文前 16 字节
        │           ├── 截去尾部 8 字节填充
        │           └── 解析 JSON → LogEntry
        │
        └── LogRepository.BatchSave(featureID, entries)
              │
              ├── 按月分组 → 每组 500 条批量插入
              ├── INSERT IGNORE + enc_data_hash (MD5) 去重
              └── 更新 user_daily_stats 汇总表
```

### 5.2 增量同步

```
SyncService.SyncFeatureIncremental()
  │
  ├── 查询 sync_state 表，获取每个 enabled feature 的 last_log_time
  ├── 对每个 feature:
  │     │
  │     └── SyncFeature(featureID, last_log_time, now)
  │           │
  │           └── 从断点继续拉取
  │
  └── 同步完成后更新 sync_state:
        ├── last_log_time = 本次最大 log_time
        ├── total_synced += 本次成功数
        └── last_sync_at = 当前时间
```

### 5.3 定时调度

```
SchedulerService.Start(interval)
  │
  └── goroutine 中运行 ticker:
        │
        └── 每个 tick 调用 SyncService.SyncAllFeaturesIncremental()
              │
              ├── 记录 sync_history（触发方式 = "scheduler"）
              └── 更新各 feature 的 sync_state
```

### 5.4 解密管道

```
加密数据: enc_key (base64) + enc_data (base64)

Step 1: RSA 解密
  input  = base64_decode(enc_key)
  output = RSA.DecryptPKCS1v15(rsa_private_key, input)
  result = 16 字节 AES 密钥

Step 2: AES 解密
  iv         = enc_data 前 16 字节（base64 解码后）
  ciphertext = enc_data 第 17 字节起（base64 解码后）
  plaintext  = AES-CBC-128Decrypt(key=AES密钥, iv=iv, data=ciphertext)
  plaintext  = pkcs7_unpad(plaintext)
  plaintext  = plaintext[0 : len(plaintext)-8]  // 截去尾部 8 字节

Step 3: JSON 解析
  parsed_json = json.Unmarshal(plaintext)
```

### 5.5 密钥热切换

```
密钥存储结构:
  keys/
  ├── v1/
  │   └── rsa_private_key.pem    ← 文件权限 600
  ├── v2/
  │   └── rsa_private_key.pem

rsa_key_versions 表:
  ┌────────┬───────────┬─────────┬──────────────┬────────────┐
  │ id     │ version   │ is_active │ private_key_path │ activated_at │
  ├────────┼───────────┼─────────┼──────────────┼────────────┤
  │ 1      │ v1        │ false   │ keys/v1/...  │ 2024-01-01 │
  │ 2      │ v2        │ true    │ keys/v2/...  │ 2024-06-15 │
  └────────┴───────────┴─────────┴──────────────┴────────────┘

解密策略:
  1. 优先使用 is_active=true 的版本
  2. 解密失败时，遍历所有历史版本回退尝试
  3. RSADecryptor 实例按版本号缓存在内存中（map[string]*RSADecryptor）

密钥文件加密:
  - 若设置 KEY_ENCRYPT_KEY 环境变量（hex 编码的 32 字节密钥），
    PEM 文件使用 AES-256-GCM 加密存储
  - 未设置时 PEM 文件明文存储
```

### 5.6 数据去重与分表

**去重策略**：
- `enc_data_hash` 字段存储 `enc_data` 的 MD5 哈希
- 唯一索引 `uk_dedup (feature_id, log_time, enc_data_hash)`
- 批量插入使用 `INSERT IGNORE`，重复记录静默跳过

**分表策略**：
- 表名格式：`log_{feature_id}_{YYYYMM}`
- 按 feature ID 和日志月份自动建表（`CREATE TABLE IF NOT EXISTS`）
- 已存在的表自动补列和索引（`ALTER TABLE ADD COLUMN`，忽略已存在错误）

**动态表结构**：
```sql
CREATE TABLE log_{feature_id}_{YYYYMM} (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  feature_id INT NOT NULL,
  log_time BIGINT NOT NULL,
  idc VARCHAR(32),
  enc_data TEXT,
  enc_key TEXT,
  raw_json TEXT,
  parsed_json JSON,
  enc_data_hash CHAR(32),
  login_openid VARCHAR(64) AS (JSON_UNQUOTE(JSON_EXTRACT(parsed_json, '$.login_user.openid'))) VIRTUAL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_log_time (log_time),
  INDEX idx_feature_logtime (feature_id, log_time),
  UNIQUE KEY uk_dedup (feature_id, log_time, enc_data_hash),
  INDEX idx_login_openid (login_openid)
);
```

`login_openid` 是 MySQL 5.7+ JSON 虚拟列，从 `parsed_json.login_user.openid` 提取，用于手机号快速匹配查询。

### 7.4 Axios 封装

```typescript
// api/index.ts 核心逻辑
const api = axios.create({
  baseURL: '/api/v1',
  timeout: 30000,
})

// 请求拦截器：自动附加 Bearer token
api.interceptors.request.use((config) => {
  const token = localStorage.getItem('token')
  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }
  return config
})

// 响应拦截器：
//   1. 自动解包 response.data
//   2. 401 时清除本地令牌并刷新页面（跳转登录）
api.interceptors.response.use(
  (response) => response.data,
  (error) => {
    if (error.response?.status === 401) {
      localStorage.removeItem('token')
      localStorage.removeItem('username')
      window.location.reload()
    }
    return Promise.reject(error)
  }
)
```

## 8. 安全设计

### 8.1 认证机制

- **JWT 签名算法**：HS256
- **令牌有效期**：24 小时
- **密码存储**：bcrypt 哈希（`bcrypt.DefaultCost`），验证时使用 `bcrypt.CompareHashAndPassword`
- **单账号体系**：用户名和密码从配置文件/环境变量加载，不支持多用户

### 8.2 登录限流

- 基于 IP 的简单限流器
- 同一 IP 1 分钟内最多 5 次登录失败
- 触发限流后该 IP 被锁定 1 分钟
- 内存中维护，后台 goroutine 每 5 分钟清理过期记录

### 8.3 密钥文件安全

- 启动时递归检查 `keys/` 目录下所有 `.pem` 文件权限
- 若权限包含 group/other 读写位（`perm & 0077 != 0`），自动修正为 `600`
- 可选：通过 `KEY_ENCRYPT_KEY` 环境变量启用 PEM 文件 AES-256-GCM 加密存储

### 8.4 操作审计

- `OperationLog` 中间件对所有 API 请求自动记录
- 记录内容：用户、操作分类、HTTP 方法、路径、状态码、耗时、IP
- 异步写入（goroutine），不阻塞请求响应
- `/health` 端点排除审计

### 8.5 TLS

- 系统本身不处理 TLS
- 生产环境建议在 Nginx 前部署反向代理（如 HAProxy / Nginx 外层）处理 TLS 终止
- 政务微信 API 通信可配置 `WEWORK_INSECURE_SKIP_VERIFY` 跳过证书验证（仅用于私有化部署场景）

## 9. 已知限制

| 限制 | 说明 | 影响 |
|------|------|------|
| QueryService 内存分页 | 跨多表查询后在 Go 内存中合并排序再切片 | 数据量大时有 OOM 风险 |
| 单管理员账号 | 无 RBAC，仅一个管理员账号 | 不适合多人使用场景 |
| 日志表原生 SQL | LogRepository 使用原生 SQL（动态表名无法用 GORM） | 其余 Repository 使用 GORM 查询构建器 |
| 内存分页无游标 | 分页基于 offset，深页查询性能下降 | 大数据量下后页加载慢 |
| 无数据库事务跨 feature | 同步时各 feature 独立事务 | 单 feature 失败不影响其他 feature |
| token 无刷新机制 | 24 小时固定有效期，无 refresh token | 长时间操作可能中途过期 |
